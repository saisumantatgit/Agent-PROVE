Write the T1 Validator agent — the pre-execution approach validator.

This agent is the ORCHESTRATOR that:
1. Receives a PROPOSED APPROACH (numbered steps) and CONTEXT (what exists, what's been tried, key numbers)
2. Launches N framework agents (default 5, configurable) in PARALLEL against the approach
3. Collects all framework verdicts
4. Synthesizes a final verdict: VALIDATED / VALIDATED_WITH_ADJUSTMENTS / REJECTED

The T1 agent prompt structure (from Mission 1086):
```
You are T1 — the approach validator for {{MISSION_NAME}}.
Your job is to stress-test this proposed approach using {{N}} thinking frameworks.

## The Proposed Approach (validate this):
{{APPROACH — numbered list}}

## Context
{{What exists, what's been tried, key numbers, relevant files}}

## Apply {{N}} Frameworks
For each framework, dispatch the corresponding agent from .claude/agents/:
- first-principles → fundamental truth check
- inversion → failure mode analysis
- five-whys → root cause of the problem statement
- occams-razor → simplicity test
- via-negativa → removal test
(+ any additional frameworks selected)

## Synthesis
After all framework agents return:
1. Count PASS vs FAIL verdicts
2. Identify CRITICAL findings (any single finding that invalidates the approach)
3. Identify ADJUSTMENT findings (improvements that don't invalidate)
4. Produce final verdict:
   - VALIDATED: All frameworks PASS, no critical findings
   - VALIDATED_WITH_ADJUSTMENTS: No critical findings, but adjustments recommended
   - REJECTED: Any critical finding, or majority FAIL
5. Output a risk register with all findings ranked by severity

## Output Schema
{
  "mission": "{{MISSION_NAME}}",
  "verdict": "VALIDATED | VALIDATED_WITH_ADJUSTMENTS | REJECTED",
  "frameworks_applied": N,
  "framework_results": [
    {"framework": "name", "verdict": "PASS|FAIL", "findings": [...], "evidence": [...]}
  ],
  "critical_findings": [...],
  "adjustments": [...],
  "risk_register": [{"risk": "...", "severity": "high|medium|low", "mitigation": "..."}]
}
```

Key design decisions:
- Default 5 frameworks, but accept --frameworks N to use more/fewer
- Frameworks are selected by name — user can specify which ones
- If not specified, use: first-principles, inversion, five-whys, occams-razor, via-negativa
- All framework agents run in PARALLEL (they're independent)
- A single CRITICAL finding from ANY framework = REJECTED

Write the Cycle Auditor agent — the per-cycle quality auditor.

This agent is dispatched DURING execution (after each cycle/phase completes) to audit the output:
1. Receives: CYCLE_OUTPUT (what was just produced), CYCLE_CONTEXT (what was expected), INPUT_FILES (files to verify)
2. Launches N framework agents (configurable) against the cycle output
3. Collects framework verdicts
4. Enforces the Evidence Protocol on all output
5. Applies Orchestrator Accountability checks

The cycle auditor adds these checks ON TOP of framework analysis:

**Evidence Protocol Enforcement:**
- Every claim in CYCLE_OUTPUT must cite a source_file + location
- Every number must have a derivation chain
- Flag any uncited inference as EVIDENCE_VIOLATION

**Orchestrator Accountability:**
- Verify output file exists and is valid (JSON parseable if JSON)
- Verify entry count matches expectations
- Verify no null/empty required fields
- Cross-check claims against actual file contents

**Escalation Logic:**
- All frameworks PASS + no evidence violations → CYCLE_APPROVED
- Any framework FAIL but no critical → CYCLE_APPROVED_WITH_WARNINGS (list warnings)
- Any evidence violation → CYCLE_FAILED (must fix and re-audit)
- 3 consecutive CYCLE_FAILED → ESCALATE (stop and ask user)

Output schema:
{
  "cycle": "{{CYCLE_NAME}}",
  "verdict": "CYCLE_APPROVED | CYCLE_APPROVED_WITH_WARNINGS | CYCLE_FAILED | ESCALATE",
  "framework_results": [...],
  "evidence_violations": [...],
  "accountability_checks": {
    "output_valid": true/false,
    "count_matches": true/false,
    "no_empty_fields": true/false,
    "claims_verified": true/false
  },
  "warnings": [...],
  "consecutive_failures": N
}
