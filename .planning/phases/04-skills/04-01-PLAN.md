Create the validate-approach skill with proper YAML frontmatter and workflow.

Frontmatter:
- name: validate-approach
- description: Validate a proposed approach using multiple thinking frameworks before execution
- Arguments: APPROACH (required, the approach to validate), --frameworks N (optional, default 5), --framework-names (optional, comma-separated list)

Skill workflow:
1. Parse the user's approach description (from arguments or prompt)
2. Gather context: ask user for relevant files to include as INPUT_FILES
3. Configure frameworks: use --frameworks N to select how many, --framework-names to specify which
4. Dispatch t1-validator agent with the approach, context, and framework configuration
5. Present the T1 verdict to the user in a clear format:
   - Overall verdict (VALIDATED / VALIDATED_WITH_ADJUSTMENTS / REJECTED)
   - Per-framework results (expandable)
   - Risk register (sorted by severity)
   - Recommended adjustments (if any)
6. If REJECTED: ask user to revise approach and re-validate
7. If VALIDATED_WITH_ADJUSTMENTS: present adjustments, ask user to accept/modify

The skill should handle the case where the user provides a plan file path instead of inline text — read the file and extract the approach.

Include usage examples in the skill:
- `/validate "Parse all XML files, extract metrics, classify by type"`
- `/validate @.planning/phases/01/PLAN.md --frameworks 3`
- `/validate "My approach to building the API" --framework-names first-principles,inversion,occams-razor`

Create the think-cycle skill with proper YAML frontmatter and workflow.

Frontmatter:
- name: think-cycle
- description: Apply thinking framework agents to audit a cycle's output during execution
- Arguments: SUBJECT (required, what to analyze), --frameworks N (optional, default 5), --input-files (optional, paths to read), --expected-count (optional, for accountability check)

Skill workflow:
1. Parse the subject (what to analyze — could be a problem statement, cycle output, or file path)
2. If subject is a file path, read it and use as CYCLE_OUTPUT
3. Configure frameworks (same as validate-approach)
4. Dispatch cycle-auditor agent with subject, context, and framework configuration
5. Present results:
   - Overall verdict (CYCLE_APPROVED / CYCLE_APPROVED_WITH_WARNINGS / CYCLE_FAILED / ESCALATE)
   - Evidence protocol results (any violations highlighted)
   - Accountability check results (output valid, counts match, no empty fields, claims verified)
   - Per-framework findings
   - Warnings (if any)
6. If CYCLE_FAILED: present specific failures, suggest fixes
7. If ESCALATE: stop execution, present full failure history, ask user for guidance

Include usage examples:
- `/think "Is our classification of these 50 items correct?" --input-files data/output.json`
- `/think @data/cycle3_output.json --frameworks 3 --expected-count 1086`
- `/think "Why are 456 items unclassified?" --framework-names five-whys,first-principles`
