## 1. Objective

Build the Thinking Framework Orchestrator as a reusable Claude Code plugin, extracted from Mission 1086's multi-framework validation methodology. The plugin should provide structured thinking validation (pre-execution), per-cycle auditing (during execution), and evidence-based accountability (post-execution) for any project. Ship with 8+ built-in frameworks, slash commands, and marketplace-ready packaging.

## 3. Variance Analysis

### 3.1 What the Agent Got Wrong

- **Domain leakage**: One EBA reference slipped into the `examples/audit-evidence-example.md` output. The example was supposed to be domain-agnostic but contained a regulatory reference from the ProSure context where the methodology originated. Caught during final review.
- **Package.json placement**: Initial package.json was placed at repository root instead of inside `.claude-plugin/`. This was correct per Claude Code plugin conventions, but the initial instinct was to put it inside the plugin directory. Resolved during scaffold phase.
- **Framework count drift**: Early commits referenced "8 frameworks" then "9" then "14" as agents were added. The README and CLAUDE.md had to be updated multiple times to keep counts synchronized.

### 3.2 What the Human Got Wrong

- **Over-engineering ADRs**: Initially proposed 12 ADRs covering everything from file naming conventions to testing strategy. This was pruned to 4 core ADRs plus 1 added later (ADR-005 for command journey). The Keep/Merge/Prune evaluation correctly identified that 8 of the 12 were either obvious decisions or better captured in PRODUCT.md as "Won't Do" borders.
- **Missing Gemini platform**: Did not think about Gemini CLI support until prompted. The plugin was built for Claude Code, Cursor, Codex, and OpenCode, but Gemini CLI was only added after the human was prompted about it. This should have been in the original brief.
- **Wrong-platform feedback**: Some early review feedback was oriented toward LangGraph/Python agent patterns rather than Claude Code plugin conventions. This was a context-switching issue from reviewing GAI (GordonAI) patterns that use a different orchestration model.

### 3.3 Environmental

No significant environmental issues. Local development only, no CI/CD failures, no dependency issues. The plugin is pure markdown/JSON with a single shell script, so the build environment was trivially stable.

## 6. Lessons Learned

### Sustain (Keep Doing)

- **Parallel agent dispatch (8-12 at peak)**: Framework agents are independent by design, so dispatching them in parallel was both safe and fast. This pattern should be the default for any "apply N independent lenses" task.
- **Keep/Merge/Prune for evaluating proposals**: When 12 ADRs were proposed, the Keep/Merge/Prune framework correctly reduced them to 4+1. This is the right evaluation pattern for any "should we do all of these?" decision.
- **Phase-aware defaults**: Automatically selecting the right frameworks for the current phase reduces cognitive load. This pattern (context-aware defaults with manual override) should be applied to future plugin features.
- **Validation script as quality gate**: `scripts/validate-structure.sh` catches structural issues before they ship. Having a programmatic check for plugin conventions prevents "works on my machine" drift.
- **Research BEFORE building**: The competitive landscape analysis (Research 001) identified critical gaps (.claude-plugin manifest, hooks) that shaped the v1.1.0 build. External intelligence should always precede major build work.

### Improve (Do Better Next Time)

- **Run validation script earlier**: The validation script was written during v1.1.0 but should have been the first artifact in v1.0.0. Build the quality gate before building the thing it validates.
- **External research timing**: Research 001 was done after v1.0.0 was already built. While it still informed v1.1.0, doing the competitive scan before any code would have saved the initial package.json placement issue and caught the manifest requirement earlier.
- **Count synchronization**: Framework counts drifted across README, CLAUDE.md, and agent dispatcher as new agents were added. A single source of truth (e.g., a generated count from the agents/ directory) would prevent this.
- **Example domain scrubbing**: Examples should be written from scratch for the plugin context, not adapted from real project outputs where domain-specific terms can leak through.

### Stop (Do Not Repeat)

- **Over-proposing ADRs**: 12 ADRs for a v1.0 plugin is over-engineering. The pruning to 4+1 was correct, but the time spent proposing and evaluating the other 8 was waste. Start with "what decisions are genuinely controversial?" not "what decisions exist?"
- **Wrong-platform feedback**: Reviewing patterns from LangGraph/Python agent frameworks and applying them to Claude Code plugin conventions causes confusion. Each platform has its own idioms; do not cross-pollinate review feedback without explicit platform context.
- **Retroactive planning artifacts**: The planning docs (BRIEF.md, ROADMAP.md, phase plans) were committed after the v1.0.0 build was already complete. Planning should be committed first, then the build. The timestamps tell the real story.
